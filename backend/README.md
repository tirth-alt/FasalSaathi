# FasalSaathi Backend

Voice-first farmer AI assistant — backend API. This is the **first slice**: a Hono
server with a health check, Supabase-Auth-based authentication (Google OAuth via
Supabase), and the farmer profile ("info page") API, backed by Supabase
(Postgres + Auth) with Row Level Security and app-layer Aadhaar encryption.

- Framework: **Hono** on Node (TypeScript, ESM).
- Data/Auth: **Supabase** (Postgres + Auth + Storage). Two sign-in paths:
  - **Google OAuth (primary):** the client signs in and gets a Supabase JWT;
    signup is implicit (the farmer row is created on first authenticated request).
    OAuth is **not** hand-rolled.
  - **Email/password:** `POST /auth/signup` + `POST /auth/login` issue a Supabase
    session. Signup uses the admin API with `email_confirm: true` so **no
    confirmation email is needed** — a deliberate DEMO choice (production would
    require verification).
  In both cases the backend verifies the JWT and resolves/creates the farmer row
  keyed by the auth user id.
- Sensitive data: Aadhaar is encrypted at rest with AES-256-GCM; only the last-4
  digits are ever exposed.

## Prerequisites

- Node >= 20 (verified on Node 25), npm.
- Supabase CLI is used via `npx supabase` (not installed globally).
- Docker Desktop — **only** needed for the local Supabase stack
  (`npx supabase start`). Not needed for running the server, tests, typecheck,

#### Build note

The codebase uses `.ts` extension imports and the `@/*` path alias, which the
base `tsconfig.json` enables (`allowImportingTsExtensions: true`). Plain `tsc`
emit cannot rewrite the `@/*` alias to runnable relative paths, so this project
**runs the TypeScript directly via `tsx`** in both dev and start (no compile
step), and `npm run build` performs a type-check as the CI gate. If a bundled
`dist/` is needed later, add a bundler (e.g. `tsup`/`esbuild`) that resolves the
path alias — `tsconfig.build.json` is retained as a starting point.

## What the USER must configure

This backend needs a Supabase project and Google OAuth set up. You do this once
in the Supabase + Google Cloud dashboards:

1. **Create a Supabase project** at https://supabase.com/dashboard.
   - Settings → API → copy **Project URL** → `SUPABASE_URL`.
   - Settings → API → copy **anon public** key → `SUPABASE_ANON_KEY`.
   - Settings → API → copy **service_role secret** key →
     `SUPABASE_SERVICE_ROLE_KEY`. This key bypasses RLS — keep it server-side only,
     never ship it to the Flutter/Next.js clients.

2. **Enable Google OAuth** (so users can sign in):
   - In **Google Cloud Console** → APIs & Services → Credentials → create an
     **OAuth 2.0 Client ID** (Web application).
   - Add the authorized redirect URI:
     `https://<your-project-ref>.supabase.co/auth/v1/callback`
   - In **Supabase Dashboard** → Authentication → Providers → **Google**: paste the
     Google **Client ID** and **Client Secret**, and enable it.
   - The client apps use Supabase's Google sign-in to obtain a JWT; this backend
     only verifies that JWT.

3. **Set `AADHAAR_ENC_KEY`** in `.env` to `openssl rand -base64 32`. Losing this
   key means losing the ability to decrypt stored Aadhaar ciphertext.

## Applying the database schema

The migration creates the `farmers` table, the `updated_at` trigger, and RLS
policies (a farmer can only read/update their own row).

**Hosted project (no Docker):**
```bash
cd backend
npx supabase link --project-ref <your-project-ref>
npx supabase db push       # applies supabase/migrations/* to the hosted DB
```

**Local stack (requires Docker running):**
```bash
cd backend
npx supabase start         # boots local Postgres + Auth (Docker)
npx supabase db reset      # applies migrations + runs supabase/seed.sql
```

> The seed (`supabase/seed.sql`) inserts FAKE demo users/farmers for local dev and
> references `auth.users` ids. **Do not run the seed against production** — real
> users come from Google sign-ins. No real Aadhaar appears anywhere; seeded
> `aadhaar_last4` values are placeholders and `aadhaar_enc` is null.

## Security / compliance notes

- Aadhaar is sensitive under India's **Aadhaar Act 2016** and the **DPDP Act
  2025**: it is encrypted at rest (AES-256-GCM), only the last-4 is exposed, and
  it is never logged. This is a hackathon DEMO — there is **no real UIDAI
  integration**.
- The **service-role** Supabase client bypasses RLS by design and is used only
  after a caller's JWT has been verified, scoped to that verified `auth.uid()`.
- The **user-scoped** client (anon key + the caller's JWT) is provided for the
  RLS-respecting path and is what the RLS integration test exercises.

## API contract

See [`docs/api.md`](./docs/api.md) for the full contract — request/response shapes,
headers, and error responses. This is what the Flutter and Next.js teams build
against.

| Method | Path | Auth | Purpose |
|---|---|---|---|
| GET | `/health` | public | health check |
| POST | `/auth/signup` | public | email/password signup → session |
| POST | `/auth/login` | public | email/password login → session |
| GET | `/me` | **auth** | authenticated farmer profile (safe shape) |
| POST/PUT | `/me/profile` | **auth** | create/update own profile |
| GET | `/mandis/nearby?lat=&lng=&limit=` | public | nearest mandis (F1/F2 mandi set) |
| GET | `/prices/history?commodity=&mandi_id=&days=` | public | 5-day modal trend per mandi (F1) |
| POST | `/decision` | **auth** | hold-or-sell decision with forecast range (F2) |
| GET | `/warehouses?district=&lat=&lng=` | public | nearest curated storage (F2 storage leg) |

The three reference-data endpoints (`/mandis/nearby`, `/prices/history`,
`/warehouses`) are **public** — non-sensitive market data, needed at onboarding.
`/decision` is **auth-protected** (it reads the farmer's location).

## Data / fixture + repository design (price/mandi/warehouse)

The price-trend + hold/sell endpoints read **in-app fixtures** (`src/data/*.ts`)
behind repository interfaces (`src/lib/repositories.ts`:
`MandiRepository` / `PriceRepository` / `WarehouseRepository`, with
`Fixture*` implementations). Routes depend only on the interfaces and the repos
are injected through `AppDeps` — so the data source is swappable without touching
routes or tests.

- **Why fixtures (not a live DB):** Docker is unavailable here (no local Postgres),
  and the historical daily-price dataset is still being sourced. Fixtures are the
  guaranteed-demo source. Prices are **deterministically generated** (seeded), so
  the demo is stable across restarts and tests assert exact shapes.
- **Future-swap path (documented, not on the critical path):** the Supabase
  migration `supabase/migrations/*_price_reference.sql` defines `mandis` +
  `daily_prices` (public-read RLS, service-role write), and `supabase/seed_prices.sql`
  is **auto-derived from the same fixtures**. When a real daily-price dataset (or
  Agmarknet snapshot) is sourced, a `Db*Repository` implements the same interface
  and reads from these tables — **no route changes**. Tests/routes never depend on
  the DB.

## Forecast design (ForecastProvider v0 → v1)

The forecast is **explainable, not a black box** (spec §2). `src/lib/forecast.ts`
defines a swappable `ForecastProvider` interface; the v0 implementation,
`SeasonalTrendForecastProvider`, blends two transparent signals:

1. **Seasonal index** — expected % change by horizon keyed off the current month
   (post-harvest months trend up). Encoded as a small stand-in table (documented as
   a placeholder for the CEDA-derived 5–7yr index, which we lack in this env).
2. **Recent momentum** — the 30-day price slope scaled to the horizon, blended at
   0.5 weight with a ±8% cap, so a sharp current move tilts the seasonal baseline
   without dominating it.

Output is **always a range** (`low_pct ≤ expected_change_pct ≤ high_pct`) plus
human-readable `drivers[]` and a `confidence` level — never a bare number. The
store-vs-sell arithmetic (`src/lib/decision.ts`) is a pure function mirroring the
on-device math, using eNWR constants `LTV 0.70`, interest `0.10/yr`, storage
`₹20/quintal/month`.

**v1 (TODO, behind the same interface):** a `TrainedForecastProvider` (gradient
boosting / time-series on historical daily mandi prices; features = recent
multi-mandi window + crop + month/seasonality + weather/arrivals) plugs in via
`AppDeps` with zero route changes once a dataset is sourced.

## Testing

```bash
npm test
```

- Unit tests run with a **mocked Supabase client** and **no database**:
  crypto round-trips, auth middleware (401s, load-or-create), email/password auth
  routes (signup → session + farmer row, duplicate-email 409, weak/invalid body
  400, login success, bad-creds 401, response never contains the password), profile
  routes (safe shape, encryption, validation), and an authenticated POST→GET
  round-trip proving the response never contains the plaintext Aadhaar or
  `aadhaar_enc`. Live e2e against a real Supabase Auth project is **not** run here
  (no hosted project / Docker configured); the fake covers the route logic.
- `rls.integration.test.ts` is **self-guarded**: it is skipped unless
  `SUPABASE_DB_URL` is set. When run against a real DB (plus
  `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `RLS_TEST_JWT_A`, `RLS_TEST_USER_B_ID`),
  it proves user A cannot SELECT or UPDATE user B's row.
